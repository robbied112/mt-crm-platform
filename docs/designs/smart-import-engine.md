## Problem Statement

The current import pipeline uses rule-based column matching (`semanticMapper.js`) that fails on real-world distributor reports. Files with pivot-table structure, weekly column pairs, hierarchical subtotals, offset data, and multi-sheet layouts produce garbage mappings. Users upload reports from distributor portals (VIP, SGWS, Republic National) and QuickBooks, and the system can't understand them.

Meanwhile, competitors like Claude CoWork take raw files and produce working BI dashboards instantly. The import intelligence IS the product — if data doesn't flow intelligently into dashboards, nothing else matters.

## Vision

Drag 7 files onto CruFolio. Within 3 seconds, see a beautiful summary card for each — the system identified what each one is, what it contains, and where the data will go. A single "Import All" button. 10 seconds later, every dashboard is populated. Revenue by channel from QuickBooks. Distributor health from velocity reports. Inventory days-on-hand. 85 new placements in CRM. 412 at-risk accounts flagged.

The user thinks: "This tool was built for someone exactly like me."

### Key Decisions (CEO Review)

1. **Separate `comprehend.js`** (not added to `ai.js`) — different concern (report understanding vs column matching)
2. **Claude Sonnet** (not Haiku) — comprehension quality is critical, cost is acceptable (~$0.006/file)
3. **Hybrid extraction** — structured spec for known patterns + sandboxed code generation escape hatch for unknown
4. **Client-side parallel** — `Promise.allSettled()` for multi-file comprehension (not server-side batch)
5. **DataImport.jsx stays as orchestrator** — extract sub-components, don't create new ImportOrchestrator
6. **Banner with confirm** for auto-connection of existing data — smart but respectful
7. **Feature-flagged** behind `tenantConfig.features.smartImport`

### Eng Review Refinements

8. **Tool use (structured output)** — comprehendReport uses Claude `tool_use` with JSON schema, not free-form JSON. Eliminates malformed JSON class of errors.
9. **Smart sampling** — Send first 20 + middle 20 + last 10 rows (not first 50). Better representation of hierarchical/pivot files.
10. **Three callables** in `comprehend.js`: `comprehendReport` (per-file analysis), `extractWithCode` (sandboxed code-gen), `generateIntegrationPlan` (multi-file AI correlation)
11. **Client-side extraction** for structured specs (95% case). Server-side `extractWithCode` Cloud Function only for code-gen escape hatch.
12. **useReducer** for multi-file state in DataImport (not 10+ useState calls)
13. **AI integration plan** — `generateIntegrationPlan` is an additional Claude call, not a client-side lookup table
14. **DataImport/ directory** — Extract into directory structure before adding smart import code (TODO-099)
15. **parseFile multi-sheet** — Extend with optional `{ sheets }` param (TODO-100)
16. **PR split** — PR 1: TODO-099+100+090+091+092+096. PR 2: TODO-093+094+095.
17. **Accept cold starts** — No minInstances on Cloud Function. First import may be 8-12s.

### ExtractionSpec Schema

```
ExtractionSpec {
  headerRow: number          // which row contains column headers (0-indexed)
  dataStartRow: number       // where data begins (often headerRow + 1)
  skipPatterns: [             // rows to filter out
    { column: number, pattern: string }  // e.g., {column: 0, pattern: "^Total$"}
  ]
  columnOffset: number       // shift all column references by N
  pivot: {                   // optional — for weekly/monthly column pairs
    startCol: number
    endCol: number
    groupSize: number        // columns per group (e.g., 2 for qty + revenue)
    labelRow: number         // row containing group labels (e.g., week dates)
    fieldNames: string[]     // names per column in group (e.g., ["qty", "revenue"])
  }
  sheets: string[]           // which sheet names to process (default: active)
  columnMapping: {           // maps output field names to source column indices
    [fieldName]: number
  }
  codeGen: string | null     // if non-null, use extractWithCode Cloud Function
}
```

## Security Mitigations

1. **Prompt injection defense:** XML delimiters (`<file_data>`) separating instructions from user data
2. **Code sandbox:** `vm` module in Cloud Function, no `require`/`process`/`fs`, 5s timeout, all code logged
3. **Rate limiting:** Max 20 comprehension calls per tenant per hour, max 20 files per batch
4. **Response validation:** All column indices validated against actual grid dimensions

## Rollback

Feature-flagged. Toggle `tenantConfig.features.smartImport = false` → instant rollback to existing rule-based pipeline. No data model changes. No migration needed.
